# 实时采集页：保存后与启动状态相关问题纪要

**日期**：2026-05-09  
**范围**：平台前端实时采集（`/collect/realtime`）作业保存、路由刷新、脚本启动与界面状态同步。

---

## 1. 背景

实时采集流程中，用户在保存一条采集数据后需要继续下一条 episode。平台侧通过 PATCH 更新作业进度，并用路由参数（含 `reset`）触发 WebSocket 与界面重置。与此同时，作业在数据库中的 `status`、采集端 Agent 心跳里的 `collect_running`、以及前端本地 `status`（`IDLE` / `RUNNING` / `DONE`）三者语义并不完全一致，容易产生「界面看起来像已在采集，但用户并未再次点击开始」的现象。

---

## 2. 今日主要问题

### 问题 A：保存后主按钮短暂不可点后，界面自动进入「采集中」

**现象**：用户完成保存后，「开始采集」因保存后冷却/硬门禁暂时不可点；尚未再次点击「开始采集」，界面却已显示「采集中」（例如进度 0%、无有效日志等）。

**原因（归纳）**：

1. **作业状态语义**：保存成功且未凑满目标条数时，后端仍将作业标为 `RUNNING`，表示「作业仍在进行中、还需继续采集」，**并不等价于「当前脚本进程正在录制」**。
2. **路由触发二次加载**：保存成功后会 `router.replace` 更新查询参数（含 `reset`、`t` 等），`Realtime.tsx` 中依赖 `[taskId, jobId, searchParams]` 的 `load()` 会再次执行。
3. **误判同步逻辑**：`load()` 在 `foundJob.status === 'RUNNING'` 且当前 URL 不是 reset 场景时，会拉取在线 Agent，若心跳里 `collect_running === true`，则执行 `setStatus('RUNNING')`，把界面推向「采集中」。
4. **竞态与陈旧心跳**：保存流程本地已将界面设为 `IDLE` 并清理日志，但紧随其后的 `load()` 仍可能读到 **尚未与「脚本已停」对齐的心跳**，从而覆盖为 `RUNNING`。

**结局（解决方法）**：

- 在 **`handleSaveDirect`** 成功且准备「下一条」时，写入 **`suppressHeartbeatRunningSyncUntilRef`**（截止时间 = 当前时间 + **`POST_SAVE_SUPPRESS_HEARTBEAT_RUNNING_MS`**，当前为 **12000ms**）。
- 在 **`load()`** 的「作业 `RUNNING` + 非 reset」分支中：若当前时间仍处于上述抑制窗口内，**强制保持 `IDLE`**，**不再根据 `collect_running` 自动升为采集中**。
- 用户明确要开始采集时（**`handleStartClick`**、**`handleRecollect({ autoStart: true })`**）将抑制引用 **清零**，避免影响正常的「刷新页 / 心跳恢复」场景。

**涉及文件**：`src/features/daq-editor/pages/Realtime.tsx`

---

### 问题 B：取消「正在向采集端下发启动指令…」单独状态展示

**现象**：脚本启动请求尚未返回成功前，界面展示专用文案（如「正在向采集端下发启动指令，请稍候。未收到成功响应前不会进入『采集中』。」），产品与交互希望去掉这一中间态展示。

**原因**：此前使用 **`scriptLaunchPending`**（及同步 ref）在 UI 上分叉展示标题、副文案与主按钮文案，与常规「等待中」区分开。

**结局（解决方法）**：

- 移除对 **`startingScript` / `hintStartingScript`** 的界面引用，并从 **`zh-CN.ts`、`en.ts`、`sv.ts`、`types.ts`** 中删除对应文案键。
- 用 **`scriptStartInFlight`** 仅在 **`startCollection` 全程**（含发起 `/api/script/start` 的请求窗口）**禁用相关按钮**，避免连点重复启动；**展示层仍按当前 `status` 渲染**（成功前仍为 `IDLE` 下的常规等待文案）。
- **不变**：仅在 **`/api/script/start` 返回成功** 后才会 `setStatus('RUNNING')`，逻辑上与「未确认成功不进采集中」一致，只是不再用那段说明性文案。

**涉及文件**：

- `src/features/daq-editor/pages/Realtime.tsx`
- `src/lib/i18n/messages/zh-CN.ts`、`en.ts`、`sv.ts`
- `src/lib/i18n/types.ts`

## 3. 相关机制备忘（未在本次文档中逐一展开）

下列项与「保存 / 启动 / 假 RUNNING」风险相关，已在前后端其它迭代中部分落地，可作为联调时的检查清单：

- **保存后冷却与硬门禁**：`POST_SAVE_COOLDOWN_MS`、`POST_SAVE_HARD_BLOCK_MS`（当前均为 **1000ms**），减轻保存后立即「开始采集」与 WS / 路由重置的竞态。
- **RUNNING 后无 WS 输出看门狗**：长时间无任何脚本输出时请求 stop 并回到 `IDLE`，缓解「界面采集中但实际未跑」的假状态。
- **采集启动串行（单进程）**：后端 `agent_collect_proxy` 等对同一 Agent / 输出维度使用 **asyncio.Lock**，减轻并发启动导致的路径或序号冲突（多 worker 部署时需另行分布式锁）。

---

## 4. 验证建议

1. **问题 A**：保存一条数据后，在冷却期内观察界面应维持「待采集 / 等待中」语义，**不应**在未点击「开始采集」时因刷新作业与心跳而直接进入「采集中」。
2. **问题 B**：倒计时结束发起启动后，进度标题与副文案**不出现**「下发启动指令」类专用句；启动成功后才进入「采集中」；启动失败仍应有终端错误信息与回到 `IDLE`。
