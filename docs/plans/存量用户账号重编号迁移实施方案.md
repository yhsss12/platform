# 存量用户账号重编号迁移实施方案

## 一、当前存量 account_id 现状核对（基于代码与库结构）

### 1.1 历史行为

- **013 迁移**对 `account_id` 为空时执行 `UPDATE users SET account_id = username`，使存量用户曾用「展示名/旧登录名」作为登录账号。
- **014 迁移**仅将「启用中、最早创建」且 `account_id IN ('Pibot','admin')` 的超管改为 `Pibot0001`，**未**对 ADMIN/OWNER/USER 等做团队流水号重编号。
- **新建用户**走 `allocate_team_scoped_account_id`，格式为 `{teams.code}{seq:04d}`，与存量回填路径不一致。

### 1.2 结论（定性）

| 类型 | 典型现象 |
|------|-----------|
| 超级管理员 | 收口后应为 `Pibot0001`（若已执行 014 且数据符合条件）。 |
| 其余角色 | 多数仍为 **旧 `username` 或历史字符串** 作为 `account_id`，除非曾手工改库或走新创建逻辑。 |
| 是否「只有超管对、其余全旧」 | 在**未做过本次迁移**的环境下，**是常见状态**；具体须用 SQL 在目标库核对（见自测）。 |

### 1.3 建议核对 SQL（迁移前在目标库执行）

```sql
-- account_id 仍等于 username 的非超管
SELECT id, account_id, username, role::text
FROM users
WHERE UPPER(TRIM(role::text)) <> 'SUPER_ADMIN'
  AND account_id = username;

-- 已是「非空 code + 四位数字」形态（仅供参考，不代替业务验收）
SELECT id, account_id, username FROM users
WHERE UPPER(TRIM(role::text)) <> 'SUPER_ADMIN'
  AND account_id ~ '^[A-Za-z0-9_]+\d{4}$';
```

## 二、非超管用户「所属团队」的确定方式

### 2.1 数据依据（与业务库一致）

- **主依据**：`team_users`（普通成员）与 `team_admins`（团队管理员）的 **并集**。
- **不单独以** `projects` / `project_members` 作为账号归属团队：账号规则按「团队维度」流水号，与列表页「团队」口径一致；用户若仅存在于项目侧而无团队行，视为**未挂靠团队**，须运维先补 `team_users` 或 `team_admins` 后再迁移。

### 2.2 多团队时的 canonical 团队

- 取该用户在所有关联团队 ID 中的 **字典序最小** `team_id`（与「稳定、可重复」一致；与列表里多团队合并展示不冲突，账号只对应一个编码团队）。

### 2.3 团队编码 `teams.code`

- 表 `teams` 上存在 **`code` 非空** 与 **`uq_teams_code` 唯一约束**（见 `backend/app/models/team.py`）。
- 若某团队 `code` 异常为空，迁移会失败并提示，需先清洗数据。

## 三、存量用户如何按 team_code + 四位流水号重编号

### 3.1 范围

- **包含**：`role` 经规范化后 **非 `SUPER_ADMIN`** 的全部用户（含 ADMIN、OWNER、USER 及历史 `ADMINISTRATOR`、`MEMBER` 等仍落在 `users` 表中的行）。
- **排除**：超级管理员（不参与团队流水）。

### 3.2 组内顺序（稳定、可重复）

- 按 **canonical 团队**分组。
- 组内用户排序：**`created_at` 升序（NULLS LAST）**，再 **`id` 升序**。
- 分配：`{teams.code}{序号:04d}`，序号从 **1** 起连续。

### 3.3 唯一性与并发冲突避免

- **两阶段更新**：先将待迁用户的 `account_id` 改为临时值 `_mig_{无连字符的 user id}`，释放旧唯一键；再写入最终 `team_code+四位`。
- **全局唯一**：最终账号集合与超管 `Pibot0001` 及临时阶段未触碰的行无冲突（超管不参与第一阶段临时键）。

### 3.4 `team_account_counter` 同步

- 对每个涉及团队：`next_seq = GREATEST(当前 next_seq, 本团队本次分配的最大序号)`，保证后续 `allocate_team_scoped_account_id` 不会撞号。

---

## 四、超级管理员固定为 Pibot0001

- 重编号**不修改**超管的 `account_id`（不参与团队流水）。
- 迁移末尾再次执行与 **014 同类**的兜底：将「启用中、按 `created_at` 最早的一条超管」的 `account_id` 设为 `Pibot0001`；若其他用户误占 `Pibot0001`，先迁出到临时 `_rel_` 形式（与 014 一致思路）。
- **`username` 可不改为 Pibot**（产品允许展示名保持）。

---

## 五、登录与 JWT 影响

1. **迁移后登录**：登录键为 **`account_id`**；旧 **`username` 不能再作为登录账号**（除非人为把某账号设成与某 `username` 相同，应禁止）。
2. **已颁发 Token**：Access/Refresh 中 **`sub`（或等价主体）绑定 `account_id`**，变更后旧 token **无法对应到新 `account_id`**，应视为 **失效**；需 **重新登录**。此为 **预期安全行为**。
3. **避免半套状态**：在维护窗口 **先跑完数据库迁移**（本 revision），再发布/重启应用；迁移期间避免并发创建用户。

## 八、自测方案

1. **备份** `users`、`team_account_counter`（及可选 `refresh_tokens` / `auth_sessions`）。
2. **迁移前**：执行上文核对 SQL；确认无非超管且**零团队关联**用户，否则迁移应 **失败** 并打印 id 列表。
3. **执行**：`cd backend && alembic upgrade head`（或等价流水线）。
4. **迁移后**：
   - 非超管 `account_id` 均为 `{code}{四位}`，且 **`account_id <> username`**（除非刻意同名）。
   - `users.account_id` 唯一约束无冲突。
   - `team_account_counter.next_seq` 不小于各团队已用最大序号。
   - 超管：`Pibot0001` 唯一启用主体符合预期。
5. **登录**：抽样用 **新账号 + 原密码** 登录；旧用户名登录失败。
6. **前端**：用户管理页账号列与用户名分离。

---

## 九、特别强调（验收标准）

- 超管固定 **`Pibot0001`**（兜底逻辑）。
- 所有非超管账号 **`teams.code` + 四位流水**，库内真实更新，而非仅前端映射。
- **`username` 仅展示名**，不改为登录账号。
- 孤儿用户必须先补团队关系，否则迁移拒绝执行。
