# 《存量用户账号重编号迁移方案》

本文档基于当前仓库代码与 Alembic 迁移链，说明「用户管理页账号列仍为旧字符串」的根因、存量如何收口到 **team_code + 4 位流水号**（即数据库字段 `teams.code` + `0001`…）、超管 **Pibot0001** 固定规则，以及登录与 JWT 风险。  
**字段说明**：业务口径中的「team_code」与模型/表字段 **`teams.code`** 一致，下文统一写 **`teams.code`**。

---

## 一、当前问题定位

### 1.1 现象

- 超级管理员在用户管理页「账号」列已为 **`Pibot0001`**（与 014/启动逻辑一致时）。
- 其他用户「账号」列仍像 **旧用户名或任意旧字符串**，与「用户名」列容易相同或极像，**未呈现 `TM120001` 这类格式**。

### 1.2 根因（代码级结论）

**主要原因在数据层历史回填，而非 `/users` 列表“绑错字段”。**

| 环节 | 行为 | 结论 |
|------|------|------|
| Alembic **013** `013_user_account_id_and_counters.py` | 新增 `account_id` 后执行：`UPDATE users SET account_id = username WHERE account_id IS NULL OR account_id = ''` | **存量非超管用户的 `account_id` 被直接设为当时的 `username`**，与最终规则「账号 ≠ 展示名」相悖，也是页面上「账号列像旧用户名」的直接来源。 |
| Alembic **014** | 超管账号收口为 `Pibot0001` 等 | **仅处理超级管理员相关冲突与固定账号**，不批量重写其他用户的 `account_id`。 |
| Alembic **015** `015_renumber_non_super_account_ids.py` | 对**非超管**按所属团队 `teams.code` + 团队内序号生成 `code0001`、`code0002`… | **若未执行成功或未部署到该环境 head，013 留下的「account_id = 旧 username」会一直保留**。 |
| `GET /users` + `UserOut` | 返回数据库中的 `account_id` | **接口字段映射正确**；若仍显示旧值，**应优先怀疑数据库未跑通 015 或 015 前置条件不满足**。 |

### 1.3 对三个排查问题的明确结论

**（1）存量普通用户 / 管理员 / 负责人的 `account_id` 是不是旧值？**

- **在未成功执行 015 的环境下：可以明确为「是」——至少经历过 013 的用户，其 `account_id` 曾与 `username` 被设为一致（或后续手工改过但仍非 team 流水号）。**
- **014 只保证超管侧 `Pibot0001` 等逻辑，不保证其他人已是 `teams.code` + 四位流水。**
- **是否「只有超管被处理成 Pibot0001、其他人仍是 username → account_id」**：在典型「只升到 014、未到 015」或「015 失败回滚」场景下，**是**。

**（2）迁移逻辑是否只处理了「新增用户」？**

- **存量**：靠 **013 回填 username**；**真正按团队规则重编号的是 015**，与「创建用户」代码路径无关。
- **新增**：`POST /users` 中，若创建时带 **`team_id`**（或团队管理员创建，会绑定其管辖团队），则调用 `allocate_team_scoped_account_id`，按 **`teams.code` + `team_account_counter` 递增** 生成账号，**符合 team_code + 4 位流水**。
- **例外（与最终规则冲突）**：超级管理员创建 **OWNER/USER** 且 **未选 `team_id`** 时，走 `allocate_displaced_account_id`，当前实现生成的是 **`Pibot` + 四位数字**（与超管账号命名空间相邻），**不是 `teams.code` + 四位**。这与「除超管外一律 team_code+流水」的锁死规则不一致，属于**产品/代码路径缺口**，需在规则收口时单独处理（见下文「实施后代码建议」）。

**（3）`/users` 列表是否已正确返回 `account_id`？**

- **是**：列表项来自 `User` 模型字段 `account_id`，schema 为 `UserOut` / `UserListItemOut`，前端用户管理页账号列绑定 `user.account_id`。
- **若页面仍显示旧字符串，应判定为库内 `account_id` 仍为旧值**，而不是列表接口映射错误。

## 二、最终账号规则（与迁移目标对齐）

1. **超级管理员**：全系统唯一，**`account_id` 固定 `Pibot0001`**；`username` 可为展示名（如 `Pibot`），**不用于登录**。
2. **管理员 / 负责人 / 普通用户**：**`account_id` = `teams.code` + 4 位十进制流水**（例：`TM120001`）。**不允许**再以旧 `username` 或随意字符串作为登录账号。
3. **`username`**：仅展示名，可与账号不同；**迁移只改 `account_id`，不改 `username`**。

## 三、存量用户如何按 teams.code + 四位流水重编号

### 3.1 仓库内已有实现：015

迁移 **`015_renumber_non_super_account_ids`**（`backend/alembic/versions/015_renumber_non_super_account_ids.py`）已实现：

- **范围**：`role` 经 `UPPER(TRIM(role::text)) <> 'SUPER_ADMIN'` 的**所有非超管**（含历史 `ADMINISTRATOR`/`MEMBER` 等若仍存在）。
- **团队归属**：对每个用户，在 **`team_users` ∪ `team_admins`** 中取 **`ORDER BY team_id ASC LIMIT 1`** 的一个 `team_id`（**多团队用户只取字典序第一个团队**参与编号，需在运维上知晓）。
- **组内顺序**：同一 `team_id` 内按 **`created_at ASC, id ASC`** 排序，第 *i* 个用户分配 **`{teams.code}{i:04d}`**（从 0001 起）。
- **唯一性**：先统一改为临时 `_mig_{uuid}`，再写入新 `account_id`，避免与 `users.account_id` 唯一约束冲突。
- **计数表**：更新 **`team_account_counter.next_seq`** 为「该团队已分配人数」与现有值取较大者，避免新用户创建时流水号与存量冲突。
- **超管**：末尾调用 `_ensure_super_admin_pibot0001`，保证活跃超管占用 **`Pibot0001`**，并处理误占。
- **硬前置**：任一非超管用户若 **不在 `team_users` 且不在 `team_admins`**，迁移 **直接 `RuntimeError` 中止**并列出 `user_id`，需先补团队关系再跑。

### 3.2 与「流水连续、唯一、不冲突」的说明

- **连续**：015 在**每个团队内**按创建时间重排后为 `0001…n`，满足「组内连续」。
- **全局唯一**：依赖 `users.account_id` 唯一索引；015 在分配后校验无重复。
- **与已生成新账号冲突**：两阶段更新 + 全量规划后再写入，可避免中途撞车；若 `teams.code` 导致某账号等于 `Pibot0001`，015 会显式报错（防与超管冲突）。

### 3.3 多团队、无团队等边界

| 场景 | 015 行为 | 建议 |
|------|-----------|------|
| 用户只属于一个团队 | 使用该团队 `code` 编号 | 正常 |
| 用户属于多个团队 | 仅 **`team_id` 最小**的那条关系参与编号 | 若业务要求「主团队」非字典序最小，需事先调整 `team_users`/`team_admins` 数据或改迁移策略（本次方案以现有脚本为准） |
| 非超管但无任何团队关系 | **迁移失败** | 先补 `team_users` 或 `team_admins`，或业务上确认是否应删/停用该用户 |
| `teams.code` 为空 | **迁移失败** | 先清洗 `teams` 表 |

## 四、超级管理员固定 Pibot0001

- **014** + **`get_or_create_admin_user`**（`backend/app/crud/user.py`）+ 015 末尾 **`_ensure_super_admin_pibot0001`** 共同保证：**启用中的超管**最终占用 **`Pibot0001`**。
- **不参与**按团队流水重排；**不修改**其 `username` 作为迁移目标（除非你们单独有展示名规范）。

---

## 五、`/users` 列表接口是否需要调整

- **不必为「显示真实 account_id」再改接口**：已返回 `account_id`、`username`，列表另有 `team_name`/`team_id` 聚合。
- **迁移完成后**：列表自然展示新账号；搜索 `_user_search_filter` 已对 **`account_id` 与 `username` 的 ilike** 生效，**「按账号或用户名搜索」在迁移后仍然可用**（用户需知晓**登录账号已变**，应用新 `account_id` 搜）。

## 六、登录与 JWT 影响（风险控制）

### 6.1 存量用户改账号后如何登录

- **结论**：迁移完成后，用户**必须使用新的 `account_id` 登录**；**`username` 不能作为登录名**（与当前登录接口设计一致：登录键为 `account_id`）。
- **需对最终用户/运维明确告知**：这是一次**登录账号变更**，不是改密码。

### 6.2 已登录 Token

- **JWT `sub`（或等价主体）绑定 `account_id`** 时，**旧 token 在改库后无法对应到用户行**（校验按新 `account_id` 查库），等效于 **立即失效**。
- **结论**：这是**预期安全行为**；无需为「保留旧 sub」做兼容（否则与重编号目标矛盾）。可选：**迁移窗口内短暂维护公告**，建议用户重新登录。

### 6.3 避免「半套状态」

1. **维护窗口**：停写或只读 → **先在同一环境跑完 Alembic 到 head（含 015）** → 验证抽样用户 `account_id` → 再发布应用。
2. **不要**先上线依赖新账号逻辑的前端、而后台库仍是 013 回填数据。
3. **015 失败**时：**不提交部分用户更新**（脚本为事务内执行则全有或全无；若以单条 UPDATE 执行需确认连接是否 autocommit——请以实际迁移执行方式为准，生产建议在事务中执行并先备份）。
4. **Refresh token / session**：若服务端存储了 `account_id`，迁移后应视为失效，用户重新登录即可。

## 七、实施后代码与产品建议（非本次「仅方案」必选，但为锁死规则）

为与「**除超管外一律 teams.code+四位**」完全一致，建议在后续迭代中：

1. **创建用户**：超级管理员创建 **OWNER/USER** 时 **强制 `team_id`**，或去掉 **`allocate_displaced_account_id`** 用于非超管的路径，避免再出现 **`Pibot0002` 式**平台账号。
2. **文档/注释**：将 `allocate_displaced_account_id` 的用途收缩为「仅迁移/冲突临时占用」或废弃，避免与超管 `Pibot0001` 命名空间混淆。

（**说明**：上述为规则锁死后的代码收口；**存量重编号本身已由 015 覆盖**。）

## 九、自测方案

1. **迁移前**：在测试库执行 `SELECT id, account_id, username, role FROM users WHERE UPPER(role::text) <> 'SUPER_ADMIN' LIMIT 20`，确认存在 `account_id = username` 或不符合 `code####` 的行。
2. **补数据**：确保每个非超管用户在 `team_users` 或 `team_admins` 中至少有一条；`teams.code` 非空。
3. **执行**：`alembic upgrade head`，确认无 015 报错。
4. **迁移后**：
   - 超管：`account_id = 'Pibot0001'`。
   - 每团队：按 `created_at` 顺序，`account_id` 为 `{code}0001`、`{code}0002`…。
   - `username` 与 `account_id` 不要求相等。
5. **接口**：`GET /users` 抽样检查 JSON 中 `account_id`。
6. **登录**：用**新 `account_id` + 密码**登录成功；用**旧账号字符串（若曾等于 username）**登录应失败。
7. **搜索**：用户管理页搜索框输入新 `account_id` 片段、以及 `username`，均能筛到对应用户。
8. **Token**：迁移前拿到的 access token 在迁移后访问需登录接口应失败或 401，重新登录后正常。

## 十、总结表

| 问题 | 结论 |
|------|------|
| 非超管为何仍显示旧账号？ | 013 将 `account_id` 回填为 `username`；**015 未成功执行或环境未升到 head** 时会一直如此。 |
| `/users` 是否有问题？ | **无映射问题**；问题在**库内 `account_id` 值**。 |
| 存量如何收口？ | **运行并打通 015**；前置补全团队关系与 `teams.code`。 |
| 超管？ | **固定 `Pibot0001`**，014 + 015 兜底。 |
| 登录与 JWT？ | **新账号登录**；**旧 token 失效为预期**。 |
