# 《用户管理页账号列与所属团队列修正方案》

## 一、当前问题定位

### 1. 「账号」「用户名」两列内容几乎相同

- **后端**：`GET /users` 已通过 `UserOut.model_validate(u)` 返回 `account_id` 与 `username`（`backend/app/schemas/user.py`、`routes/users.py`）。
- **前端**：表格已绑定 `user.account_id` 与 `user.username`（`src/app/(platform)/admin/users/page.tsx`），**并非表头改了、数据仍绑 username**。
- **根因（数据层）**：迁移 `013` 将 `account_id` 回填为 **与历史 `username` 相同**；超管引导逻辑仍把 **`account_id` 与 `username` 都写成 `Pibot`**。因此在「展示名未单独改过」的用户上，**两列自然相同**；超管在库中多为 `account_id = username = Pibot`，与产品「账号 `Pibot0001`、用户名可为 `Pibot`」不一致。

### 2. 「账号列仍像旧用户名」的体感

- 在 **`account_id` 与 `username` 相同** 时，用户会误以为账号列仍是展示名。
- 收口要点：**数据上**保证超管 `account_id = Pibot0001`、`username` 可为 `Pibot`；列表接口继续显式返回两字段，前端无需再猜。

### 3. 缺少「所属团队」

- **当前**：列表项无 `team_name` / `team_id`，前端无法展示团队。
- **补法（最小增量）**：在 `GET /users` 聚合页内用户与 `team_users` / `team_admins` 的 `team_id`，再查 `teams.name`，为每条记录增加 **`team_name`（及可选 `team_id`）**。

---

## 二、`/users` 列表接口返回结构核对

| 字段 | 当前 | 修正后 |
|------|------|--------|
| `id` | ✓ | ✓ |
| `account_id` | ✓ | ✓（超管经迁移/引导为 `Pibot0001`） |
| `username` | ✓ | ✓（展示名，可与账号不同） |
| `role` / `is_active` / `created_at` / `last_login_at` | ✓ | ✓ |
| `team_name` | ✗ | ✓ 超管固定 **「平台」**；团队用户为团队名称（多团队用 ` / ` 拼接） |
| `team_id` | ✗ | ✓ 可选：仅单团队时给主 `team_id`，多团队为 `null` |

**响应模型**：使用列表专用项 `UserListItemOut`（在 `UserOut` 基础上扩展 `team_name`、`team_id`），避免强迫单用户 `UserOut`  everywhere 带团队字段。

---

## 三、前端「账号 / 用户名」列映射核对

- **账号列**：应绑定 `account_id`（已正确）；收口后随数据变为 `Pibot0001`、`TM01xxxx` 等即可区分。
- **用户名列**：绑定 `username`（已正确）。
- **新增**：**所属团队**列绑定 `team_name`，超长用 `title` + `maxWidth` + 省略样式。

---

## 四、所属团队信息如何补到列表接口

1. `list_users` 在得到当页 `users` 后，收集 `user_id` 列表。
2. 在 **`assets_db`** 中：
   - 查 `TeamUser`、`TeamAdmin` 得到 `user_id -> team_id` 集合；
   - 批量查 `Team` 得 `team_id -> name`。
3. 组装规则：
   - **`role == SUPER_ADMIN`**：`team_name = "平台"`，`team_id = null`。
   - **否则**：无关联 → `team_name = "—"`；单团队 → `team_name = 团队名`，`team_id` 可填该 id；多团队 → `team_name = "名A / 名B"`，`team_id = null`（避免歧义）。

---

## 五、超级管理员 `Pibot0001` 显示规则收口

| 位置 | 规则 |
|------|------|
| **数据库** | 迁移将**当前启用中的超管**（若 `account_id` 仍为 `Pibot`/`admin`）更新为 **`Pibot0001`**；非超管若误占 `Pibot0001` 先迁出。 |
| **`get_or_create_admin_user`** | 正账号统一为 **`Pibot0001`**；新建/迁移时 **展示名默认 `Pibot`**（`username`），与登录账号分离。 |
| **`/auth/me`、顶栏** | 继续 **`username` 主展示**（可为「Pibot」）；**账号语义**在账号设置页等已用 `account_id`。 |
| **`GET /users`** | **账号列**即 `account_id`，应为 **`Pibot0001`**；**所属团队**为 **「平台」**。 |

## 七、自测方案

1. **超管**：库中 `account_id` 为 `Pibot0001`，`username` 可为 `Pibot`；用户管理页 **账号**=`Pibot0001`，**用户名**=`Pibot`，**所属团队**=`平台`。
2. **团队用户**：账号形如 `TM010001`，用户名为创建时填写；**所属团队**为真实团队名（多团队见拼接）。
3. **超管新建管理员**：刷新列表后三列均正确。
4. **团队管理员新建负责人/用户**：同上。
5. **搜索**：仍支持按账号或用户名模糊（后端已有 `account_id`/`username` OR 条件）。

---

## 八、对登录与团队账号生成的影响

- **团队账号生成**（`team_code`+流水）：**不变**。
- **登录**：超管需使用 **`Pibot0001`** 登录（若此前用 `Pibot`，迁移后需切换）；JWT `sub` 随 `account_id` 更新。
